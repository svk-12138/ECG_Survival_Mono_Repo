# 卒中论文数据处理操作手册

这份手册是给医生或学生使用的数据整理说明。  
它解决的问题很直接：把你手里的“标签表 + XML 心电文件”，整理成训练可以直接读取的标准文件。

整理成功后，训练会更简单，因为训练脚本可以自动读取整理后的路径配置。

如果你已经完成数据整理，下一步请看：

- [docs/卒中论文训练操作手册.md](卒中论文训练操作手册.md)

## 这份手册适合什么情况

下面这些情况，都建议先做这一步：

- 你手里有卒中队列标签表
- 标签表里有结局时间和结局事件
- ECG 原始文件是 XML
- 同一个患者可能有多次 ECG 检查
- 你希望后续训练时不用再手工一条一条对数据

## 先理解一下：为什么要先做数据整理

这一步不是“多此一举”，它主要解决 4 个问题：

1. 把标签表和具体 ECG 文件一一对应起来
2. 保留同一患者的多次 ECG，不会误删
3. 按 `patient_SN` 分组切分数据，避免同一患者同时出现在训练集和验证集里
4. 提前检查每份 XML 到底支持 8 导还是 12 导，避免训练时才报错

对于只有 1200 条数据的卒中课题，这一步尤其重要，因为样本少，更要避免数据泄露和格式错误。

## 你需要提前准备什么

### 1. 一份标签表

标签表可以来自：

- CSV
- Excel 另存后的 CSV

最少必须有下面 4 列：

- `patient_SN`
- `event`
- `time`
- `xml_file`

这些字段的通俗解释如下：

- `patient_SN`：患者编号，用来做分组，防止同一患者的数据泄露到不同集合
- `event`：结局是否发生，通常写 `0` 或 `1`
- `time`：随访时间或结局时间
- `xml_file`：这条 ECG 对应的 XML 文件名，或相对路径

### 2. 一个 XML 文件夹

这个文件夹里放你的 ECG XML 原始文件。

### 3. 一个输出文件夹

比如：

- `D:\stroke_project\processed`

这个文件夹用来保存整理后的标准结果。

## 建议你把文件整理成这样

下面只是一个示例，目录名字可以按你自己的习惯改：

```text
D:\stroke_project
├─ labels.csv
├─ XML
│  ├─ 0001.xml
│  ├─ 0002.xml
│  └─ ...
└─ processed
```

如果你的数据结构大致像这样，后面的命令会更容易照抄。

## 最短操作方法

在仓库根目录运行下面这个命令即可：

```powershell
python scripts/prepare_survival_dataset.py `
  --labels "D:\stroke_project\labels.csv" `
  --xml-dir "D:\stroke_project\XML" `
  --output-dir "D:\stroke_project\processed"
```

如果你用的是 Linux、macOS 或 WSL，可以写成：

```bash
python scripts/prepare_survival_dataset.py \
  --labels /path/to/labels.csv \
  --xml-dir /path/to/XML \
  --output-dir /path/to/processed
```

## 运行前，先逐项对照一下

在按回车之前，请确认下面 3 件事：

1. `--labels` 指向的是标签表
2. `--xml-dir` 指向的是 XML 文件夹
3. `--output-dir` 指向的是你想保存结果的位置

只要这 3 个路径写对，脚本就会开始整理数据。

## 整理成功后会生成什么

输出目录通常会生成下面这些文件：

- `manifest.json`
- `training_inputs.json`
- `处理报告.txt`
- `process_report.json`
- `audit\lead_audit.csv`
- 其他异常明细 CSV

你可以这样理解它们：

- `manifest.json`：训练真正要读的清单文件
- `training_inputs.json`：给训练脚本自动读取的参数文件
- `处理报告.txt`：最适合医生先看的中文结果摘要
- `process_report.json`：更详细的机器可读报告
- `lead_audit.csv`：每份 XML 的导联检查结果

## 哪几个文件最值得优先看

如果你只想快速判断“这批数据能不能训练”，优先看下面 3 个：

- `processed\处理报告.txt`
- `processed\manifest.json`
- `processed\training_inputs.json`

如果你还想判断应该用 8 导还是 12 导，再看：

- `processed\audit\lead_audit.csv`

## 这一步和训练是怎么衔接的

这是最关键的一点：

如果 `processed\training_inputs.json` 存在，训练脚本 `scripts/train_stroke_thesis.ps1` 会自动优先读取下面这些信息：

- `manifest`
- `xml_dir / csv_dir`
- `recommended_lead_mode`
- `waveform_type`

也就是说，数据整理完成后，你在训练阶段通常不需要再手工填写这些路径，只需要重点确认任务类型、导联模式、数据划分比例即可。

## 为什么项目强调按 `patient_SN` 分组

现在训练切分遵循下面的规则：

1. 优先使用 `patient_SN`
2. 如果旧数据里没有 `patient_SN`，才回退到 `patient_id`

这样做的目的，是防止同一位患者的不同 ECG 被分到不同数据集。  
否则模型可能“见过同一个人”，结果看起来很好，实际上论文结果会失真。

所以对于你的 1200 条卒中数据，强烈建议：

- 新数据统一提供 `patient_SN`
- 同一患者的多次 ECG 全部保留
- 不要在 Excel 里手工随意去重

## 8 导和 12 导是怎么判断的

处理脚本会自动审计 XML，并给出：

- `requested_waveform_supports_8lead`
- `requested_waveform_supports_12lead`
- `recommended_lead_mode`

你可以这样理解：

- 如果 `recommended_lead_mode = 12lead`，通常说明数据比较完整，可以按 12 导训练
- 如果 `recommended_lead_mode = 8lead`，通常说明当前波形类型下，不是每份 XML 都能稳定提供完整 12 导

遇到后一种情况，不建议硬上 12 导训练，应该先按 8 导完成主流程，保证数据一致性。

## 什么时候脚本会直接失败

下面这些问题出现时，脚本会停止，并且不会给出可训练的清单文件：

- 标签表缺少必要列
- `xml_file` 找不到对应 XML
- `event`、`time`、`patient_SN` 为空或格式非法
- XML 无法解析
- XML 中取不到 `PatientID`

这时不要急着改模型，先看这些文件：

- `处理报告.txt`
- `process_report.json`
- `audit\missing_xml_rows.csv`
- `audit\invalid_label_rows.csv`
- `audit\invalid_patient_rows.csv`

一般问题都能在这些文件里定位到具体哪一行数据出了问题。

## 对 1200 条卒中数据的实际建议

建议按下面顺序做：

1. 先把标签表整理成标准 CSV
2. 运行一次数据整理脚本
3. 先看 `处理报告.txt`
4. 再看 `lead_audit.csv`，确认是 8 导还是 12 导更稳妥
5. 最后再去运行训练脚本

这样做的好处是：

- 先把数据问题暴露出来
- 再决定训练用 `8lead` 还是 `12lead`
- 避免模型报错后，才回头查数据
